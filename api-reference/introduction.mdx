## Welcome to the Rails API

The Rails API is a robust, production-ready REST API designed for secure financial transaction processing. It provides endpoints for managing transactions, settlements, ledgers, and system operations with enterprise-grade authentication and error handling.

## API Overview

The Rails API is built on modern REST principles and includes:

- **Secure Authentication** - JWT Bearer token authentication for all endpoints
- **Type-Safe Operations** - Comprehensive request/response schemas
- **Real-time Settlement** - Pending transaction management and settlement operations
- **Detailed Ledger Tracking** - Complete transaction history and audit trails
- **Webhook Support** - Real-time transaction notifications
- **Health Monitoring** - System status and metrics endpoints

## Authentication

All API endpoints require Bearer token authentication via JWT. Include your authentication token in the `Authorization` header:

```bash
Authorization: Bearer YOUR_JWT_TOKEN
```

### Authentication Methods

<Tabs>
  <Tab title="Bearer Token (Recommended)">
    For most use cases, authenticate using JWT Bearer tokens:

    ```bash
    curl -X GET https://api.rails.com/api/transactions \
      -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
    ```

    **Token Format**: JWT with the following claims:
    - `sub` - User/account identifier
    - `iat` - Issued at timestamp
    - `exp` - Expiration timestamp (typically 1 hour)
    - `scope` - Permission scope (e.g., "read:transactions", "write:transactions")

  </Tab>
  <Tab title="Admin Token">
    For administrative operations, use the `x-admin-token` header:

    ```bash
    curl -X POST https://api.rails.com/api/admin/operations \
      -H "x-admin-token: admin_token_here"
    ```

    Admin tokens are separate credentials with elevated permissions.

  </Tab>
</Tabs>

### Obtaining Your Token

1. Authenticate with your credentials via the `/auth/login` endpoint
2. Receive a JWT token in the response
3. Include the token in subsequent requests as shown above

```typescript
// Example token generation
const rails = new Rails({
  serverURL: process.env.RAILS_API_URL,
  security: {
    bearerAuth: process.env.RAILS_API_TOKEN,
  },
});
```

## Request/Response Format

### Request Headers

All API requests should include:

```
Content-Type: application/json
Authorization: Bearer YOUR_JWT_TOKEN
```

### Response Format

Successful responses return HTTP 200 with JSON data:

```json
{
  "data": {
    "id": "txn_123abc",
    "status": "completed",
    "amount": 1000.00,
    "currency": "USD"
  }
}
```

Error responses include a detailed error object:

```json
{
  "error": {
    "code": "INVALID_ACCOUNT",
    "message": "The specified account does not exist",
    "details": {
      "account_id": "acct_invalid"
    }
  }
}
```

## Core Concepts

### Transactions

Transactions represent the movement of funds between accounts. Each transaction has:

- **Unique Reference** (`txn_ref`) - Immutable identifier
- **Status** - pending, completed, failed, or reversed
- **Accounts** - Source and destination
- **Amount & Currency** - Transaction value
- **Metadata** - Custom fields for tracking

### Ledger & Settlement

The ledger maintains a complete record of all transactions:

- **Pending Ledger** - Transactions awaiting settlement
- **Settlement Process** - Batches pending transactions for final processing
- **Audit Trail** - Immutable history of all changes

### Account Management

Accounts are the fundamental units in the system:

- **Account ID** - Unique identifier (`acct_xxxxx`)
- **Account Type** - Checking, savings, or virtual
- **Balance** - Current available balance
- **Status** - Active, suspended, or closed

## Rate Limiting

The API implements rate limiting to protect against abuse:

- **Default Limit**: 1000 requests per hour
- **Headers**: Check `X-RateLimit-*` headers in responses
- **Backoff Strategy**: Implement exponential backoff on 429 responses

```
X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 999
X-RateLimit-Reset: 1704067200
```

## Error Handling

The API uses standard HTTP status codes:

| Status | Meaning |
|--------|---------|
| 200 | Request succeeded |
| 400 | Bad request / validation error |
| 401 | Unauthorized (missing or invalid token) |
| 403 | Forbidden (insufficient permissions) |
| 404 | Resource not found |
| 409 | Conflict (duplicate or invalid state) |
| 500 | Server error |

Errors include detailed information to help with debugging:

```json
{
  "error": {
    "code": "INSUFFICIENT_FUNDS",
    "message": "Account has insufficient balance for this transaction",
    "statusCode": 400,
    "timestamp": "2024-01-15T10:30:00Z",
    "requestId": "req_abc123xyz"
  }
}
```

## SDK Integration

For easier integration, use the official TypeScript SDK:

```bash
npm install @rails/sdk
```

The SDK handles authentication, serialization, and error handling automatically:

```typescript
import { Rails } from "@rails/sdk";

const rails = new Rails({
  serverURL: "https://api.rails.com",
  security: {
    bearerAuth: process.env.RAILS_API_TOKEN,
  },
});

// Type-safe API calls
const transaction = await rails.transactions.create({
  fromAccount: "acct_123",
  toAccount: "acct_456",
  amount: 1000,
  currency: "USD",
});
```
